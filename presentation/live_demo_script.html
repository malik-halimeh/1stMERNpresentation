Live Demo Script — Click by Click

🛒 OptiCart Appliances

Every click, URL, and DevTools action in order, with a pre-demo checklist and per-step fallbacks. Verified against the refreshed feature inventory — no step routes through an unfinished flow.

Purpose. The exact, rehearsed click path for the live demo, in one continuous shopper story. Each step names the presenter, the action, the expected result, and a fallback. Every path here is a feature verified COMPLETE in the refreshed inventory — no step routes through anything unfinished. URLs assume defaults; confirm the ports against your .env before you start.

Pre-demo checklist (do this 10 minutes before)

  • Seed the database — run the seed script so the 3 accounts, products, orders, coupons exist. Confirm at least one product is delivered for this customer (needed for the verified-review step).
  • Both servers running — API up, client dev server up; open the storefront once to confirm it loads.
  • Logged OUT — clear the session so the demo starts as a guest. Empty the guest cart (or pre-seed one item if you want the merge to be visible).
  • DevTools ready — Chrome DevTools open on the Network tab, "Preserve log" ON, filtered to Fetch/XHR. Zoom the browser to ~110% so the audience can read it.
  • Coupons confirmed — SAVE10 (10% off ≥ $50) and FLAT50 ($50 off ≥ $500) are active.
  • Two browser profiles or windows — one for the customer, one for staff — so you don't burn time logging in and out.
  • Screen share — share the browser window (not the whole desktop); notifications silenced.
  • Backup screenshots folder open in case a live step misbehaves (see §Screenshots).
  • Terminal ready for the optional test_fixes.ts run (security slide).

Let the access token expire or trigger a refresh; show the /auth/refresh call and the retried original. User sees nothing.
Fallback: DON'T stall — describe it, point to services/api.ts, use screenshot B12. This is the one flaky step; have the screenshot ready.
13
403 RBAC rejection.
Stay logged in as the customer. In DevTools → Network, click any authenticated call (e.g. /notifications) and copy its full Authorization: Bearer … value. In the Console, call an admin-only endpoint with that customer token:
fetch('http://localhost:5000/api/audit-logs',{headers:{Authorization:'Bearer <customer-token>'}}).then(r=>console.log(r.status)) → 403. The token is valid but the role is customer, so the server's authorize('super_admin') rejects it. Say "enforced on the server, not hidden in the UI."
Why the token is required: access auth is a Bearer header, not a cookie — a bare fetch (only credentials:'include') returns 401, and simply visiting /admin is a client-side redirect, not a server 403.
Fallback: screenshot B13 of the 403 envelope.

Global fallback plan

  • A server dies mid-demo. Don't debug live — switch to screenshots and narrate; restart the server off-screen while the next presenter talks.
  • Mock payment hangs. Use screenshot B15 and move to the fulfilment queue with an existing paid order from the seed.
  • The 401→refresh won't reproduce. Never wait for it — it's timing-dependent. Describe it, point to services/api.ts, show B12. (Pre-scripted in the presentation script as an "if reproducible" step.)
  • Wrong data state (e.g. no delivered order for the review). Re-seed before the session; if it happens live, pivot to a product that already has reviews and describe the verified-purchase rule from the ERD.
  • Total demo failure. Everything demoed maps to a slide + screenshot; you can deliver the entire story from the deck alone without losing the narrative.

6Planning — Database Design (ERD)Mahmod70score
Say
This is our data model — fourteen collections, and the two main ones are User and Product. My rule for shaping the data was: if something is owned by one document and always read with it, put it inside; if it's shared or edited on its own, link to it instead. So a product's variants (its sizes and colors) live inside the product — they don't exist on their own. But an order only links to the product. Here's the important part: an order saves a snapshot — a frozen copy of the product's name and price at the moment of purchase. So if we rename or reprice the product later, old orders still show what the customer actually bought. Money is always stored as whole cents, never a decimal, because decimals can round money wrong. And every index — a database shortcut — is there for a reason: for speed, or to enforce a rule, like "one review per customer per product," which the database itself guarantees.

7Architecture & Request LifecycleMalik75score
Say
There are three layers: a React app in the browser, an Express server written in TypeScript, and a MongoDB database — plus Stripe for payments and Cloudinary for images. Every request goes through the same set of steps in order: security headers, CORS, reading the request body, rate limiting, then checking who you are and what you're allowed to do, then the actual code that answers the request, and finally one error handler that turns any failure into the same JSON shape. One detail I'm proud of: Stripe's payment confirmation is read as raw bytes before we parse the JSON. Stripe signs the raw bytes, so if we parsed them first we couldn't check the signature — and a fake payment could sneak in. Getting that order right is what makes our payments safe. And because the whole app is TypeScript, if the frontend and backend ever disagree about the data, it fails at build time instead of breaking live.

11Malik — Auth, Axios & the Global Search BarMalik110score
Say
I'll sign in as the customer. The second I do, the guest cart merges — same items, now on my account.

Now let me show you how the frontend talks to my backend. We use a small library called Axios — think of it as a smart messenger for network requests. We set it up once, and then two things happen on every request without us repeating any code. First, open the Network tab: every call carries a login token in its header, but no page adds that by hand — Axios adds it in one place. Second, that token is short-lived and kept only in memory, never in local storage, so a malicious script can't steal it. When it expires, watch: a request fails with 401, Axios quietly gets a fresh token using a secure cookie, and retries the request — the user sees nothing. We picked Axios over the browser's built-in fetch for exactly this reason: with fetch we'd copy this token-and-retry code onto every call, and one missed spot is a security hole. Axios lets us write it once.

One more security check: I'll call an admin-only endpoint as this normal customer — and it comes straight back 403 Forbidden. That's role-based access checked on the server. Hiding the button was never enough.

And on the admin side, a small thing we're proud of: the search bar. It's one component we built ourselves and reuse on every admin page — products, orders, users, audit logs. We didn't reach for a heavy search service like Algolia, because we didn't need to: our data is already in MongoDB, and one small, consistent search box that talks to our own API does the job everywhere. You'll see it in action on the admin pages coming up.

Cart's ready and I'm logged in — let me actually pay.

Expected questions
Axios or fetch — what's the real difference?
Fetch is built into the browser but bare: you set headers and check errors on every call yourself. Axios is configured once — a base URL, automatic JSON, and "interceptors" that add the token and auto-refresh on 401 for every request. For an app with lots of logged-in calls, that central control is safer and less repetitive. See client/src/services/api.ts.
Why build your own search box instead of a library?
Our searches are simple field matches over data already in MongoDB. A hosted search service (Algolia, Elasticsearch) adds cost, setup, and a second copy of the data to keep in sync — overkill here. One small reusable component keeps every admin page consistent.
Why in memory and not localStorage?
localStorage is readable by any script — one XSS and the token's gone. Memory clears on reload, and we get a new token from the http-only cookie.
What if the 401→refresh doesn't happen live?
I'll describe it and point to services/api.ts; I also have a screenshot in the backup.

15Smart Discovery — Views, Trending & RecommendationsHaya Mahmod Malik90score
Say
Haya: Remember the view counts under each product? Here's how they work. Every time someone opens a product, the app quietly records a "view." We don't save them one at a time — we collect a few and send them together in a small batch, so it's light on the server. They live in their own collection and even delete themselves after 180 days, so the data stays fresh. That view data does two jobs: the number you see on each product, and it powers "trending." Mahmod?

Mahmod: Once a day, a scheduled job looks at the last seven days. For trending, it counts which products got the most views recently and flags the top ones — so "trending" means "lots of people are looking at this right now." We also flag most-selling, but that one ranks by how many units were actually sold in orders, not views — because looking and buying are two different signals. Malik, take the recommendations.

Malik: The last one is "customers also bought." It's a co-occurrence engine — a simple version of what big stores use. Once a day it looks at what each customer bought; every time the same person bought two products, those two get a point together, and more recent purchases count a little more. The pairs with the highest scores become each product's recommendations. We work all this out overnight and just read the saved result on the page, so the product page stays fast. And the super-admin dashboard turns the same raw data into charts — revenue, orders, and spend — using the database to add it up, never numbers we fudge in the browser.

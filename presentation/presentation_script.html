6Planning — Database Design (ERD)Mahmod70score
Say
This is our data model — fourteen collections, with User and Product as the two hubs. My core design rule was: embed what's owned and read together, reference what's shared and edited independently. So a product's variants live inside the product document — they have no life of their own — but an order only references the product. Now the important part: orders and purchases store snapshots — a frozen copy of the name and price at that moment — so if we rename or reprice a product later, history never silently changes. And money is always integer cents, never a float, because floats can't represent money exactly. Every index you see exists to serve a specific query or enforce a specific rule — like one review per customer per product, guaranteed by the database itself.

7Architecture & Request LifecycleMalik75score
Say
Three layers: a React single-page app, an Express API in TypeScript, and MongoDB, with Stripe and Cloudinary alongside. Every request runs the same chain — security headers, CORS, body parsing, rate limiting, then per-route authentication and authorization, then the controller, and finally one error handler that turns any failure into the same JSON shape. One detail I'm proud of: the Stripe webhook is mounted with the raw body before the JSON parser, because Stripe signs the raw bytes — parse them first and you can't verify the signature. That ordering isn't cosmetic; it's what makes our payments tamper-proof. The whole thing is TypeScript end to end, so a mismatch between frontend and backend is a compile error, not a 2 a.m. bug.
Do
Show the layered architecture + request-lifecycle diagrams. Trace the raw-webhook fork with the cursor.

Why one error handler?

Is the API stateless?
Per request, yes — authorization rides in the JWT. The only server state is the hashed refresh token and the in-memory rate limiter.

11Malik — Auth & Security DemoMalik100score
Say
I'll sign in as the customer. The moment I do, the guest cart merges — same items, now on my account. Now look at the Network tab. Every API call carries a short-lived access token in the Authorization header — and that token lives only in memory, never in local storage, so a malicious script can't steal it. When it expires, watch: a call 401s, the client silently refreshes using an http-only cookie, and retries — the user sees nothing. And the refresh isn't just reissued, it's rotated, with theft detection: if an old token is ever replayed, we revoke every session. One more — I'll hit an admin-only endpoint as this customer and it comes straight back 403. That's role-based access enforced on the server; hiding the button was never going to be enough.

16What We LearnedAll75score
Say
Haya: I got confident with single-flight request coordination — making ten simultaneous 401s trigger one refresh, not ten. It taught me the fix isn't more state, it's one point of coordination. And I really internalized why a token belongs in memory, not storage.
Malik: Mine is compensating transactions — MongoDB gives me single-document atomicity, but when an operation spans two documents I learned to make the second step reversible and undo the first on failure, instead of reaching for heavy machinery. That one idea made our whole costing feature safe.
Mahmod: Mine is partial indexes — a unique constraint that only applies to a filtered subset. It turned a real bug into an elegant rule: one active low-stock alert per variant, unlimited history, enforced by the database. I now see indexes as business rules, not just speed.

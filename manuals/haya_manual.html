2 · Core concepts you must own

Three contexts, one job each

AuthContext holds the user and the in-memory access token and drives silent refresh; ShopContext holds cart + wishlist and handles the guest→auth merge; ToastContext surfaces success/error toasts. Keeping them separate means a cart update doesn't re-render auth-only consumers.

One Axios instance + single-flight refresh

Every call goes through one instance (services/api.ts) with withCredentials so the httpOnly refresh cookie rides along. A response interceptor catches 401s and triggers exactly one refresh via a shared module-level promise; concurrent 401s queue on that promise and replay once it resolves. The access token lives in memory, so a reload starts tokenless and refreshes on mount.

Route guards

<ProtectedRoute> in App.tsx gates account and admin routes by role and redirects unauthenticated users to login. It's UX, not security — the server enforces the real boundary — but it keeps users out of screens they can't use.

Guest-first shopping

Cart and wishlist work with no account (localStorage via utils/guestCart.ts). On login the local state merges into the server and clears. This is why "add to cart, then sign in" never loses the cart.

URL as state

Catalog filters, sort, page, and the header search all live in the URL query string. The header search is a submit form that pushes ?search=; ProductList reads params so results are shareable and back-button-correct.

3 · End-to-end trace: browse → buy

  1. Storefront loads; AuthContext silently refreshes from the cookie; category nav is fetched live and rendered by slug.
  2. User searches — submit pushes ?search=fridge; ProductList reads the param, shows skeletons, fetches /products, renders cards with pagination.
  3. Product page: pick a variant (price/gallery/stock update), add to cart → ShopContext updates optimistically and a toast confirms.
  4. Cart page revalidates stock; a coupon field previews a discount. Checkout collects an address and runs the Stripe step.
  5. After sign-in mid-flow, the customer is returned to the storefront (never stranded on login), cart intact via merge. Order confirmation renders from the server response.

4 · Questions & model answers (25+)

Q1 · Why Context instead of Redux?
The app has three well-bounded slices of shared state (auth, shop, toast). Context covers that cleanly without Redux's boilerplate. If cross-cutting state grew, I'd reconsider, but today Redux would be overkill.
Q2 · Where do you store the access token and why?
In memory inside AuthContext — never localStorage. localStorage is script-readable (XSS risk). On reload we silently refresh from the httpOnly cookie, so users don't notice.
Q3 · What is silent refresh?
When a request 401s, the Axios interceptor calls /auth/refresh, gets a new access token, and replays the original request — the user never sees the blip.
Q4 · Ten requests 401 at once — do you fire ten refreshes?
No. A shared module-level promise makes it single-flight: one refresh, the rest await it and replay. Otherwise we'd hammer the endpoint and risk rotation races.
Q5 · Why one Axios instance?
So the baseURL, credentials, and the refresh interceptor are configured once and every call inherits them. No scattered fetch calls with inconsistent auth handling.
Q6 · How do guest carts survive login?
ShopContext keeps a localStorage cart for guests; on login it merges into the server cart and clears local storage. Stock is re-validated server-side.
Q7 · How do route guards work?
<ProtectedRoute> checks the user's role from context and either renders the route or redirects. Admin routes require staff roles; account routes require any logged-in user.
Q8 · Isn't a client-side guard insecure?
By itself, yes — anyone can edit client code. It's purely UX. The server authorises every request, so bypassing the guard just yields a 403 from the API.
Q9 · Where do you land a user after login?
Customers go to the storefront, staff to /admin/dashboard. If the redirect param points back at an auth page we ignore it, so nobody loops on the login screen.
Q10 · How does filtering/sorting/search work on the list page?
All of it is URL query state. Changing a filter updates the query string; ProductList reads params and refetches. That makes results shareable and back-button-correct.
Q11 · Why is the search a submit form now, not a live dropdown?
An earlier version had a live-suggestion dropdown; the current header search submits to the product list with ?search=. It's simpler, fully keyboard-accessible, and URL-driven.
Q12 · How do you handle loading states?
Skeleton loaders for lists and detail pages so layout doesn't jump, plus disabled buttons during mutations. The user always sees progress, never a frozen screen.
Q13 · How do variants work on the product page?
A product has embedded variants; selecting one swaps price, gallery, and stock indicator, and add-to-cart targets that variant's SKU — never the product generically.
Q14 · What's the DataTable used for?
It's the reusable admin table (sorting, pagination, row actions) behind Products, Orders, Coupons, Users, and more — one component, consistent behaviour everywhere.
Q15 · Where does SearchBox get used?
Seven admin pages (Products, Orders, Coupons, LowStock, Reviews, Users, AuditLogs). It was briefly suspected dead code; it isn't — verified by grep.
Q16 · How is the admin sidebar role-scoped?
AdminLayout filters its nav by the user's role, so an inventory manager doesn't see super-admin-only items. The server still guards each endpoint.
Q17 · How do toasts work?
ToastContext exposes success/error helpers; the Axios interceptor and mutations call them, so feedback is consistent and centrally styled.
Q18 · How does the design system stay consistent?
Tailwind tokens (colors, spacing, radii, typography) defined once in tailwind.config; components consume tokens, not hardcoded values, so a theme tweak is one place.
Q19 · Is the app responsive?
Yes — mobile-first Tailwind. The header collapses to a drawer, the admin sidebar collapses, grids reflow. A responsiveness pass fixed the earlier mobile drawer that did full reloads.
Q20 · Why React 19 and Vite?
Vite gives near-instant HMR and fast builds; React 19 is the current major. Together they keep the dev loop tight.
Q21 · How does the notification bell work?
NotificationBell shows an unread count and a dropdown; mark-all-read clears it. It reads the per-user Notification collection through the API.
Q22 · How do you show money?
The server sends integer cents; the UI formats to currency at render time only. We never do arithmetic on the displayed dollar value.
Q23 · What happens on a failed API call?
The interceptor surfaces a toast from the server's error.code/message, and the component keeps a usable state (skeleton clears, retry available). Errors are predictable because the API envelope is uniform.
Q24 · How is the profile page different from the old stub?
It's a real page (pages/admin/Profile.tsx): identity card, live name editing via PATCH, and password guidance — replacing a title-only placeholder.
Q25 · How is TypeScript helping on the frontend?
Shared response types make API mismatches compile errors, and props are typed so a wrong prop is caught before runtime. It's most valuable across the context boundaries.
Q26 · How does the category nav stay correct?
It's fetched from GET /categories and every item navigates by slug via a helper — no hardcoded names, so adding a category in admin shows up in the storefront nav automatically.

5 · Curveballs (10)

C1 · A user opens two tabs and both refresh at once. Does the UI break?
No. Client-side single-flight means each tab fires at most one refresh, and the server's 30-second grace window accepts the concurrent one without logging anyone out. The tabs stay in sync.
C2 · The refresh call itself 401s. What does the user see?
The interceptor gives up gracefully: it clears auth state and routes to login rather than looping. The half-finished request fails with a toast, not a spinner forever.
C3 · Your client guard says admin, the server says 403. Who wins and why?
The server, always. The guard can be edited in devtools; the API can't be fooled. That's why the guard is UX and the server is security — I can bypass my own guard and still get nothing.
C4 · A product's stock dropped to zero while it sat in a guest cart. Then they log in.
The merge re-validates against the server; the out-of-stock line is flagged/removed and a toast explains it. Better to show the truth at login than fail at checkout.
C5 · The 546 kB bundle — why, and what would you do?
It's currently one chunk. I'd route-split the admin area from the storefront with lazy imports and dynamic import(), since most visitors never load admin. That alone would cut the initial payload a lot.
C6 · Context re-render performance — when does it bite?
When a context value changes, all consumers re-render. I keep the three contexts narrow and memoize values so a cart tick doesn't re-render auth consumers. If a slice got hot, I'd split it or move to a selector-based store.
C7 · Someone disables JavaScript. What works?
Nothing — it's a client-rendered SPA. For SEO/first-paint I'd reach for SSR (Next.js) or prerendering the catalog, but that's a bigger architectural change than this project scoped.
C8 · How do you keep the cart from double-submitting on a slow network?
The action button disables during the in-flight request and re-enables on resolve; optimistic UI updates are reconciled with the server response, so a double click can't create two line items.
C9 · The access token expires mid-form-submit. Does the user lose their input?
No — the interceptor refreshes and replays the same request with the same body, so the submit completes transparently. The form state never leaves the component.
C10 · If you had one afternoon to improve the frontend, what?
Route-based code splitting for the admin bundle, plus a shared typed API client generated from the server types so the two can never drift. Biggest wins for effort.

6 · A new concept I learned

Single-flight request coordination. Before this project I'd have let each 401 trigger its own refresh. Learning to serialise them behind one shared promise — so N failures cause one refresh and N replays — changed how I think about async UI: the fix wasn't more state, it was a single point of coordination. It also taught me why the access token belongs in memory, not storage.

7 · Still challenging / honest gaps

  • Bundle size. 546 kB unsplit. I understand route-based splitting but haven't wired it yet.
  • No SSR. Pure CSR means weak SEO and slower first paint for the public catalog.
  • Type drift. Frontend and backend types are maintained by hand; a generated client would remove a whole class of mismatch bugs.
  • Testing. There's a backend regression suite but little automated frontend testing — component/interaction tests are the gap I'd close next.

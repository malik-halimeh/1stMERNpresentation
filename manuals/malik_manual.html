2 · Core concepts you must own

Two-token auth

Short JWT access token (15 min, memory only, carries userId+role for cheap authz) plus a long opaque refresh token (7 days, httpOnly SameSite=Strict cookie, only its SHA-256 hash in the DB). The split means: an XSS can't read the refresh token (httpOnly), and a stolen access token dies in 15 minutes.

Rotation + grace + reuse detection

Every refresh is one atomic findOneAndUpdate: match live hash → write new hash → keep previous hash for a 30-second grace window. Normal rotation, concurrent-tab-within-grace (matches prev hash, no re-rotate), and stale-token-outside-grace (theft → null all hashes → 403 AUTH_SESSION_COMPROMISED). The grace window is what stops two tabs from logging each other out.

Fail-closed webhook

express.raw() mounted before express.json() preserves the bytes Stripe signed. A missing/invalid signature is a hard 400 — there is no "trust the body anyway" branch. Mock mode is scoped to explicitly keyless dev.

Atomic money math

Stock decrement is a conditional $inc guarded by stock ≥ qty — the database, not the app, arbitrates the race. Purchases go further: one aggregation-pipeline findOneAndUpdate with $map adds stock and recomputes weighted-average cost in a single document operation, with a compensating revert if the follow-up write fails.

One error envelope

Throw AppError(msg, status, code) anywhere; a single errorHandler serialises everything to { error: { message, code } }. Predictable for the client, and no stack traces leak.

3 · End-to-end trace: a checkout

  1. Client posts the cart to POST /api/orders/checkout. authenticate attaches req.user.
  2. createCheckoutSession re-prices every line from the DB (never trusts client prices), re-validates the coupon completely, and creates a Stripe PaymentIntent (or mock).
  3. Stripe → POST /api/orders/webhook (raw body). Signature verified fail-closed.
  4. On payment_intent.succeeded: loop variants, atomic guarded $inc decrement, clamp-and-warn on lost race.
  5. Create the Order from server prices with line-item snapshots; write a purchase ProductEvent (carries userId → feeds recs); fire confirmation email; return.
  6. Any failure → typed AppError → one JSON envelope. A Stripe refund later runs before the status persists and writes a refund_decision audit.

4 · Questions & model answers (25+)

Q1 · Why two tokens instead of one?
A short-lived access token limits the blast radius of theft; a long refresh token keeps users logged in. Storing only the refresh token, httpOnly, keeps it out of JS reach. One token can't be both convenient and safe.
Q2 · Why is the access token in memory and not localStorage?
localStorage is readable by any script — one XSS and the token's gone. Memory dies on reload, and we silently refresh from the httpOnly cookie on mount, so UX doesn't suffer.
Q3 · What exactly happens on refresh?
One atomic findOneAndUpdate matching the current hash, writing a new hash, saving the previous with a 30s grace. Winner gets new tokens; a within-grace concurrent request gets an access token without re-rotating; a stale token nulls all hashes → 403.
Q4 · What is reuse detection and why does it matter?
If an old refresh token is presented outside the grace window, that's a replay — likely theft. We revoke every session for that user immediately. It turns a stolen cookie into a dead cookie fast.
Q5 · Why hash the refresh token in the DB?
Same reason we hash passwords: a database leak shouldn't hand out live sessions. We store SHA-256, compare hashes, never the raw token.
Q6 · Why SameSite=Strict on the cookie?
It stops the browser from attaching the refresh cookie to cross-site requests — a CSRF mitigation on the one credential that's automatically sent.
Q7 · How does RBAC work?
authenticate verifies the JWT and sets req.user; authorize(...roles) checks the role. The client also guards routes, but that's cosmetic — the server is the boundary.
Q8 · Why does the webhook raw-body mount come before express.json?
Stripe signs the raw bytes. If JSON parses first, those bytes are gone and the signature can't be verified. Order is correctness, not style.
Q9 · What does "fail-closed" mean here?
If we can't verify the signature, we reject (400). We never fall back to trusting the payload. An attacker can't fake a "payment succeeded" event.
Q10 · How do you prevent overselling?
The stock decrement is a conditional $inc that only applies when stock ≥ qty. The DB resolves concurrent orders; a loser clamps to 0 and logs, it never goes negative.
Q11 · Why store money in cents?
Floats can't represent 0.1 exactly — money math drifts. Integer cents is exact; we format to dollars only at the edge.
Q12 · Walk me through weighted-average costing.
New cost = (oldStock·oldCost + qty·unitCost) / (oldStock + qty). We compute it inside one aggregation-pipeline update with $map so stock and cost change together atomically — no window where they disagree.
Q13 · What if the Purchase.create fails after the stock update?
A compensating update reverts the stock/cost change. Delete does the mirror image and refuses (409) if the revert would make stock negative.
Q14 · How does refunding work?
stripe.refunds.create (mock id if keyless) runs before we persist the refunded status; we fill the refund subdoc, write a refund_decision audit, and 502 if Stripe rejects. Status never claims money moved when it didn't.
Q15 · How are errors handled consistently?
Controllers throw AppError(msg, status, code); one errorHandler serialises to {error:{message,code}}. Zod failures → 400 with detail; unknowns → generic 500.
Q16 · What does the rate limiter do?
A global 1000/15-min ceiling, plus a stricter limiter on auth routes. It's an in-memory store — fine for one instance, would need Redis to scale out.
Q17 · How does the recommendation engine work?
A nightly node-cron job computes recency-weighted product co-occurrence from purchase events and upserts top-10 lists. Requests only read the cache, with a popular-items cold-start fallback.
Q18 · Why cron and not compute on the fly?
Co-occurrence over the event history is expensive; precomputing nightly keeps request latency flat. Recommendations don't need to be second-fresh.
Q19 · How is input validated?
Zod schemas in validators/ as allow-lists — unknown fields are stripped, so a client can't set role or isActive by smuggling extra keys.
Q20 · What's in the audit log and why?
Every privileged action (role change, refund, purchase, moderation…) as an append-only entry with actor, action type, target, and before/after. 9 action types, indexed for filtering. It's accountability you can't quietly edit.
Q21 · How does the mailer behave without SMTP?
It logs the message to the console instead of sending — so demos and CI work with zero mail config. Five triggers total.
Q22 · Why is the JWT secret read lazily?
ESM hoists imports, so reading the secret at module load ran before .env was applied and always used the fallback. Reading it lazily at first use fixed that; production now refuses to boot without it.
Q23 · What happens when a customer signs in?
They land on the storefront; staff land on /admin/dashboard. A ?redirect= back to an auth page is ignored so nobody is stranded on the login screen.
Q24 · How do guest carts survive login?
Guests keep cart/wishlist in localStorage; on login the client merges them into the server cart and clears local. Stock is re-validated on read.
Q25 · What's the boot order and why does it matter?
env → Mongo (retry) → syncIndexes → middleware → routers → cron → listen. Indexes sync before traffic; the webhook raw mount precedes JSON; cron starts only once the DB is up.
Q26 · Why syncIndexes at boot?
It reconciles the live indexes with the schema — that's how the old full unique index on LowStockAlert gets dropped in favour of the partial one without a manual migration.

5 · Curveballs (10)

C1 · A refresh token is stolen. Trace what an attacker can actually do.
They get in until either the real user or the attacker refreshes and the other one replays the now-stale token — reuse detection then nulls all hashes and both are logged out. Worst case they hold a 15-min access token window. httpOnly + Strict already make the theft hard.
C2 · Two instances behind a load balancer — what breaks first?
The cron job double-runs (both instances fire the nightly recompute) and the in-memory rate-limit store splits (limits are per-process). Fix: a cron leader-lock and a Redis limiter store. Correctness of orders/payments is unaffected — those are DB-atomic.
C3 · Stripe sends the same webhook twice. Do you double-decrement stock?
The intent transition is idempotent in practice because we act on payment_intent.succeeded and create one order per intent; a repeat finds the order already exists. The honest answer: I'd add an explicit processed-event guard keyed by event id to make idempotency bulletproof.
C4 · Your access token still says role=customer but the admin just promoted them. Now what?
Nothing for up to 15 minutes — the role rides in the token. That staleness is the price of stateless authz. If we needed instant revocation we'd check a fast store (Redis) per request, trading latency for immediacy.
C5 · Why not JWTs for the refresh token too?
A JWT refresh token can't be revoked without a denylist — which reintroduces the state we were trying to avoid. An opaque, hashed, DB-backed token gives us rotation and instant revocation for free.
C6 · A coupon is validated in the preview call. Why validate again at checkout?
Because the preview response is client-controlled by the time it comes back — anyone can POST couponCode straight to checkout. Server-side re-validation (caps, expiry, min-order) is the only trustworthy gate.
C7 · Someone floods the anonymous view endpoint to inflate a product's rank. Can they?
Within rate limits, yes — view events are anonymous and unauthenticated by design (privacy + no friction). It's a known, bounded tradeoff; recommendations are advisory, not authoritative, and purchase events (which do carry userId) dominate the signal.
C8 · The mock payment mode — could it ever activate in production?
It's gated on the absence of Stripe keys. In production the secret fail-fast means a misconfigured deploy won't boot at all, which surfaces the missing-keys mistake before any traffic. I'd still add an explicit NODE_ENV==='production' assertion that keys exist.
C9 · Your error handler returns codes like COUPON_EXPIRED. Isn't that leaking logic?
It's leaking intended logic — a stable contract the client localises. What it never leaks is stack traces, query internals, or which field of a login failed (we keep auth errors generic to avoid oracle attacks).
C10 · If you had to make one thing here production-grade tonight, what?
Move the rate-limit store and a cron leader-lock to Redis, and delete server/dist from git with a proper build step. Those are the two items standing between "works great on one box" and "scales horizontally."

6 · A new concept I learned

Compensating transactions. MongoDB gives me single-document atomicity, but the Purchase flow touches two documents (Product cost/stock, then the Purchase record). I learned to treat the second write as a step that can fail and to revert the first deterministically rather than reaching for a distributed transaction. That "do, then undo on failure" pattern — and its mirror on delete — made the costing feature safe without heavyweight machinery.

7 · Still challenging / honest gaps

  • Horizontal scale. Cron and rate-limiting assume one process. I know the fixes (leader-lock, Redis) but haven't implemented them.
  • Webhook idempotency. Robust in practice, not yet bulletproof against duplicate event ids — I'd add an explicit processed-events guard.
  • File size. order.ts and purchase.ts are long; the logic is correct but would read better split into service functions.
  • Observability. morgan('dev') only — no structured logs, metrics, or tracing to actually operate this in prod.

2 · Weaknesses — don't volunteer, answer calmly if asked

W1 · Single-instance rate limiting & cron. The rate-limit store is in-memory and node-cron runs in-process, so two instances would split limits and double-run the nightly job. Answer: "Correct for one instance by design; scaling out needs a Redis limiter store and a cron leader-lock — a known, small change." (middleware/rateLimiter.ts, services/scheduler.ts)
W2 · Role staleness ≤ 15 min. A role change only bites when the access token refreshes. Answer: "Inherent to stateless JWT authz; instant revocation would mean a per-request store lookup — a latency-vs-immediacy tradeoff we chose deliberately."
W3 · Register email enumeration. A duplicate register returns AUTH_EMAIL_IN_USE. Answer: "A deliberate UX tradeoff so users know to log in instead; login errors stay generic to avoid an oracle. We could make register generic too if the threat model demanded it." (auth.ts:101)
W4 · Anonymous, spoofable view events. The batch endpoint is unauthenticated, so views can be inflated within rate limits. Answer: "By design — privacy and zero friction. Recommendations are advisory and dominated by purchase events, which are user-attributed." (routes/productEvents.ts)
W5 · Hard user deletion. Deleting a user is a hard delete. Answer: "Orders snapshot the data they need, so history survives; a soft-delete/anonymise flag would be the cleaner GDPR-friendly approach and is on the list." (user.ts deleteUser)
W6 · Webhook idempotency not bulletproof. One order per intent makes duplicates safe in practice, but there's no explicit processed-event guard. Answer: "Robust today; I'd add a processed-events set keyed by Stripe event id to make it airtight."
W7 · Dev-only logging. morgan('dev') only — no structured logs, metrics, or tracing. Answer: "Fine for development; production operability needs structured logging and metrics, which we haven't added."
W8 · No CI/CD or Docker. Answer: "The regression suite exists but isn't wired into a pipeline; CI running it on every push plus a Dockerfile is the obvious next step."

3 · Improvements — the roadmap

  1. Redis-backed rate limiting + cron leader-lock — the single change that unlocks horizontal scale. Malik
  2. CI pipeline running test_fixes.ts + both builds on every push, plus a Dockerfile and .dockerignore. Malik
  3. Route-based code splitting — lazy-load the admin bundle away from the storefront to cut the 546 kB initial payload. Haya
  4. Explicit webhook idempotency — a processed-events guard keyed by Stripe event id. Malik
  5. Soft-delete users — anonymise flag instead of hard delete. Malik
  6. Structured logging + metrics — replace morgan('dev') with JSON logs and basic request metrics. Malik
  7. Split the largest controllers — extract service functions from order.ts/purchase.ts for readability and unit-testing. Malik
  8. Generated typed API client — derive frontend types from server types so the two can't drift. Haya
  9. Frontend interaction tests — component/flow tests to match the backend's regression coverage. Haya
  10. Restore DB spec docs — the structure markdown was deleted at HEAD; keep human-readable schema docs beside the code. Mahmod
  11. Untrack server/dist — build artifact in git; add to .gitignore with a real build step. Malik

4 · Code smells — honest, and mostly minor

CS1 · server/dist tracked in git (47 files). A build artifact committed to source control — the single biggest git-hygiene drag on the audit. Fix: gitignore it and build in CI. This is the first thing an evaluator's git ls-files will surface, so know it's there.
CS2 · 600-line controllers. order.ts and purchase.ts carry a lot in one file. The logic is correct and cohesive, but they'd read better split into service functions.
CS3 · Duplicated low-stock helper (×3). The threshold/alert helper is repeated across a few call sites; it should be one shared function.
CS4 · DB structure docs deleted at HEAD. The schema is now the only source of truth; the human-readable spec files are gone. Not a code bug, but a documentation regression.
CS5 · Implicit mock-mode selection. Payment mock mode is chosen by the absence of Stripe keys rather than an explicit flag — convenient in dev, but I'd prefer an explicit PAYMENTS_MODE so intent is unambiguous.
CS6 · Git history artifacts. The log includes a brief .env leak-and-cleanup cycle and a couple of low-effort commit messages. Nothing live is exposed now, but the history isn't spotless.

None of the above is a correctness bug on the happy path. They are scalability, hygiene, and readability items — exactly the kind of self-aware list that strengthens a defense when you raise the fix before the evaluator raises the problem.

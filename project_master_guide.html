1 · What OptiCart is

OptiCart Appliances is a full-stack e-commerce application for home appliances: a customer storefront plus a role-scoped admin back office, built as a TypeScript monorepo. A shopper browses a catalog, filters and searches, builds a cart (even as a guest), checks out through Stripe, tracks orders through a seven-state lifecycle, and reviews what they bought. Staff manage products, inventory, orders, refunds, coupons, purchases, users, and analytics — every privileged action written to an append-only audit log.

Architecture
Two-package npm monorepo: client/ (React SPA) + server/ (Express API), one MongoDB database.
Backend
Node + Express 4, TypeScript, Mongoose 8, JWT auth, Stripe, Cloudinary, node-cron, nodemailer, Zod.
Frontend
React 19, Vite, React Router 7, Context API, Axios, Tailwind 3.4.
Money
Everything stored in integer cents — no floats touch money anywhere.
Roles
customer, inventory_manager, super_admin — enforced on the server, mirrored by client route guards.
Demo credentials (from README.md / server/seed.ts): customer customer@opticart.dev / Customer123! · manager manager@opticart.dev / Manager123! · admin admin@opticart.dev / Admin123!.

3 · The request lifecycle

Boot order matters (server/src/server.ts): load env → connect Mongo (with retry) → syncIndexes() → register middleware → mount routers → start cron → listen. The middleware order is deliberate:

#MiddlewareWhy it sits here
1helmet()Security headers before anything runs.
2cors() (credentials, allow-list origin)Cookies cross-origin need explicit origin + credentials.
3express.raw() on /api/orders/webhookCritical Stripe signature needs the unparsed body. Mounted before the JSON parser so the raw bytes survive.
4express.json() + cookie-parserParse everything else; read the refresh cookie.
5morgan('dev')Request logging (dev format — a noted prod gap).
6global rate limiter (1000 / 15 min)Blanket abuse ceiling; auth routes get a stricter limiter on top.
7/api/* routersThe 16 domain routers.
8404 handler → errorHandlerEverything funnels into one JSON error envelope (see §7).
The #1 thing to be able to explain: why the webhook raw-body mount comes before express.json(). If JSON parsed first, the raw bytes are gone, stripe.webhooks.constructEvent can't verify the signature, and you'd be forced to trust unverified input. This was a real bug in an earlier version; it is fixed and fail-closed now (controllers/order.ts stripeWebhook).

4 · Authentication & authorization

This is the most sophisticated part of the system and the most likely to be probed. Two tokens, two very different lifetimes and storage models:

Access tokenRefresh token
TypeJWT (signed, stateless)Opaque random string (stateful)
Lifetime15 minutes7 days
Stored whereClient memory only (never localStorage)httpOnly cookie, SameSite=Strict
In the DB?NoOnly its SHA-256 hash
CarriesuserId + role → cheap authzNothing readable — just an identifier to rotate

Rotation, grace, and reuse detection

On every POST /api/auth/refresh the server does a single atomic findOneAndUpdate: match the current hash, write a new hash, remember the previous hash with a 30-second grace window (controllers/auth.ts). Three outcomes:

  • Normal rotation — the one request that matches the live hash wins, gets a fresh access token and a rotated cookie.
  • Concurrent tab within grace — a second request arriving within 30s matches prevRefreshTokenHash; it gets a new access token but does not re-rotate. This is why two open tabs don't fight each other.
  • Stale token outside grace (theft signal) — someone replays an old token. The server nulls all of that user's refresh hashes and returns 403 AUTH_SESSION_COMPROMISED, forcing a full re-login. That's reuse detection.

Client side: single-flight silent refresh

One shared Axios instance (client/src/services/api.ts). When any request 401s, a response interceptor fires a single refresh call — a module-level promise guarantees that ten simultaneous 401s trigger one refresh, not ten (single-flight). Queued requests await that one promise, then replay. The access token lives only in memory, so a page reload starts with no token and silently refreshes on mount from the httpOnly cookie.

RBAC

authenticate verifies the JWT and attaches req.user; authorize(...roles) gates by role (middleware/auth.ts). Applied per route file — e.g. product writes require inventory_manager or super_admin; user management requires super_admin. The client mirrors this with <ProtectedRoute> in App.tsx, but the client guard is UX only — the server is the real boundary.

Honest limitation: role changes take up to 15 minutes to bite, because the role rides inside the access token and isn't re-checked against the DB until the token expires and refreshes. This is an inherent property of stateless access tokens, not a bug — say so plainly.

6 · Feature workflows

6.1 Catalog, search & filters

GET /api/products (controllers/product.ts) builds a Mongo query from allow-listed params — category slug, price range, rating, text search, sort, page — and returns items plus a meta pagination block. The storefront header search is a submit form that pushes ?search= into the URL; ProductList.tsx reads the param (no live-suggestion dropdown — that was an earlier iteration). A text index backs the search.

6.2 Product detail & variants

A product carries embedded variants, each with its own SKU, price, and stock. ProductDetail.tsx lets the shopper pick a variant (updating price, gallery, and stock state) and fires a batched anonymous view event. Add-to-cart always targets a specific variant SKU, never the product as a whole.

6.3 Cart & wishlist (guest → auth merge)

Guests get a cart/wishlist in localStorage (utils/guestCart.ts). On login, the client merges the local cart into the server cart, then clears local. The server re-validates stock on every cart read, so a cart can legitimately shrink between sessions if stock dropped. This merge-on-login pattern is worth calling out — it's why "add to cart before signing in" just works.

6.4 Checkout & payments (Stripe, fail-closed)

  1. createCheckoutSession (controllers/order.ts) re-prices the cart server-side, re-validates the coupon from scratch (existence, active, expiry, min-order, global + per-user caps — the client's preview is never trusted), and creates a Stripe PaymentIntent (or a mock in keyless dev).
  2. Stripe calls POST /api/orders/webhook with the raw body. Verification is fail-closed: a missing/invalid signature is a 400, not a fallback.
  3. On payment_intent.succeeded the handler decrements each variant's stock with an atomic conditional $inc guarded by stock ≥ qty; a lost race clamps to 0 and logs a warning instead of overselling.
  4. The order is created from server-side prices with line-item snapshots, a purchase event is recorded (this one does carry userId, feeding recommendations), and a confirmation email fires.

6.5 Order lifecycle & refunds

Orders move through a seven-state machine; updateOrderStatus enforces legal transitions and records a reason. Cancelling/refunding calls stripe.refunds.create (mock id in keyless mode) before persisting, populates the embedded refund subdoc, writes a dedicated refund_decision audit entry, and aborts with 502 REFUND_FAILED if Stripe rejects — money movement never lags the status.

6.6 Reviews (verified purchase)

Only a delivered purchase can review a product (controllers/review.ts). A unique (user, product) index enforces one review each; creating/editing/deleting runs inside a transaction that recomputes the product's denormalized ratingAvg/reviewCount, so the product card never has to aggregate reviews at read time. Client-supplied image URLs are ignored (no review upload pipeline exists).

6.7 Low-stock alerts

When a stock write crosses the threshold, a duplicate-safe alert is created. The interesting bit is the index: a partial unique index on (productId, variantSku) filtered to status:'active' (models/LowStockAlert.ts). That allows exactly one active alert per variant while still permitting many historical resolved ones — an earlier full unique index wrongly blocked the second resolve. syncIndexes() at boot drops the stale index.

6.8 Purchases & weighted-average costing NEW

The newest domain (controllers/purchase.ts, models/Purchase.ts). Recording a restock does two things in one atomic aggregation-pipeline findOneAndUpdate: it adds the quantity to stock and recomputes the product's weighted-average costPriceCents using a $map over variants. The old and new cost blend by quantity, so margins stay honest as buy-prices drift. If the Purchase.create that follows fails, a compensating update reverts the stock/cost change; deleting a purchase reverts too, refusing (409) if that would drive stock negative. Every purchase writes a stock_purchase audit entry.

6.9 Analytics, recommendations, notifications, email

  • Analytics (controllers/analytics.ts) — 5 aggregation series over delivered orders/registrations, now including purchase-spend; rendered as cards + charts in admin/Analytics.tsx.
  • Recommendations — a nightly node-cron job computes recency-weighted product co-occurrence and upserts top-10 lists (services/scheduler.ts); request-time reads are cache-only, with a cold-start fallback.
  • Notifications — per-user bell with unread count and mark-all-read (ui/NotificationBell.tsx).
  • Email — services/mailer.ts sends via SMTP if configured, else logs to console; 5 triggers (welcome/verify, reset, order confirmation, status change, low stock).

7 · Error handling pipeline

One shape for every failure. Controllers throw a typed AppError(message, statusCode, code) or call next(err); a single errorHandler (middleware/errorHandler.ts) turns anything into { error: { message, code } } with the right status. Zod validation failures become 400s with field detail; unknown errors become a generic 500 that never leaks internals. This is why the whole API is predictable to consume — and why controllers/user.ts being retrofitted onto this pipeline mattered (it used to return raw error objects).

shape of every error response
{ "error": { "message": "Coupon has expired", "code": "COUPON_EXPIRED" } }

8 · Frontend architecture

  • Three contexts. AuthContext (user + token in memory + silent refresh), ShopContext (cart/wishlist, guest merge), ToastContext (notifications). Note the README once called ShopContext "CartContext" — that drift is fixed.
  • Routing. React Router 7 in App.tsx; <ProtectedRoute> gates admin and account routes by role. After sign-in, customers land on the storefront and staff on /admin/dashboard; redirects back to auth pages are ignored so nobody is stranded on the login screen.
  • Two layouts. StorefrontLayout (two-row header, live category nav by slug, URL-synced search, footer with contact + social SVGs) and AdminLayout (role-scoped collapsible sidebar).
  • Reusable UI. DataTable, SearchBox (used by 7 admin pages), modals, skeleton loaders. Tailwind design tokens defined once in tailwind.config.

10 · Strengths & weaknesses at a glance

Strengths: refresh rotation with reuse detection; fail-closed payments; atomic money math (stock + weighted-average cost); one clean error envelope; append-only audit across 9 action types; cents-only money; verified-purchase reviews with denormalized ratings; guest→auth merge; thoughtful index strategy (partial unique, TTL, text, compound). Full detail in audit/audit.html.

Weaknesses (own them, don't hide them): single-instance rate-limit + cron; deliberate register email enumeration (UX tradeoff); anonymous spoofable view events; ≤15-min role staleness; hard user deletion; dev-only logging; no CI/Docker; server/dist tracked; 546 kB bundle; some 600-line controllers. None of these is a correctness bug in the happy path — they are scaling and hygiene items, and every one of them has a one-sentence answer ready in the question bank.

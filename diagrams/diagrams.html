1 · System architecture

Three layers. The client talks to the server over HTTPS carrying the httpOnly refresh cookie; controllers reach MongoDB, Stripe, and Cloudinary; Stripe calls back via the raw-body, fail-closed webhook.

2 · Request lifecycle & middleware order

The raw-body branch for /api/orders/webhook forks off before express.json() so Stripe's signed bytes survive. Success returns a 200 JSON body; any throw funnels into the one error envelope.

  ... --> morgan --> rateLimit --> authN/authZ --> controller
  controller -- throw AppError --> errorHandler
  controller -- success --> 200 JSON

3 · Auth: login, refresh, rotation, reuse detection

The 30-second grace window is what lets two tabs coexist; a token replayed outside it is treated as theft and nukes every session for that user.
Mermaid source · auth_flow.mmd (full sequenceDiagram)
presentation/diagrams/auth_flow.mmd
sequenceDiagram
  B->>A: POST /login (email, password)
  A->>DB: bcrypt compare · store SHA-256(refresh)
  A-->>B: JWT 15m + Set-Cookie refresh (httpOnly,Strict,7d)
  B->>A: POST /refresh (single-flight)
  A->>DB: findOneAndUpdate(new hash, prev=old, grace 30s)
  alt winner:            new access + rotated cookie
  else concurrent<=30s:  new access only (no re-rotate)
  else stale>grace:      null ALL hashes -> 403 AUTH_SESSION_COMPROMISED

4 · Product create (upload → Cloudinary → Mongo)

Files never touch disk (Multer memory → streamed to Cloudinary). The Zod allow-list strips unknown fields, and only after a successful save is the audit entry written.
Mermaid source · product_create_flow.mmd
presentation/diagrams/product_create_flow.mmd
sequenceDiagram
  A->>M: POST /api/products (multipart, files)
  Note over M: authenticate + authorize(manager|super_admin)
  M->>V: Zod allow-list (reject unknown -> 400)
  M->>CL: stream buffers to Cloudinary -> secure_url[]
  M->>DB: create Product { variants[], images[], costPriceCents }
  M->>AUD: audit product_create
  M-->>A: 201 product

5 · Checkout & payment (fail-closed, atomic stock)

Prices and coupons are recomputed server-side (the client is never trusted). The webhook is fail-closed, and stock moves via a guarded atomic $inc that cannot oversell.
Mermaid source · checkout_flow.mmd
presentation/diagrams/checkout_flow.mmd
sequenceDiagram
  C->>O: POST /checkout (cart, couponCode, address)
  O->>DB: re-price · re-validate coupon
  O->>S: create PaymentIntent (or mock)
  S->>W: webhook payment_intent.succeeded
  Note over W: verify signature FAIL-CLOSED (400)
  W->>DB: $inc stock guarded stock>=qty (clamp 0 on race)
  W->>DB: create Order (server prices, snapshots)
  W->>DB: purchase ProductEvent (userId -> recs)

6 · Database ERD (14 collections)

USER and PRODUCT are the two hubs. Snapshots on ORDER and PURCHASE keep history stable; embedded variants live inside PRODUCT; PRODUCTEVENT self-prunes via a 180-day TTL.

2 · Project structure — 7/10

Against: pages/Inventory/Categories.tsx still sits outside pages/admin/; admin/users.tsx lowercase vs sibling PascalCase; guest-wishlist helpers still live inside a page file (pages/Wishlist.tsx) and are imported by ShopContext; dev scripts (test_*.ts, update_product_images.ts) mixed into src/.

4 · Backend architecture — 9/10

For: disciplined middleware chain with a raw-body exception exactly where cryptography requires it (server.ts); per-route RBAC composition; uniform error envelope; two-phase payment→staff-confirm design; refunds that move money before persisting state; atomic single-document stock/cost math (webhook $inc guard; purchases $map weighted-average pipeline with compensating revert); side effects (audit/notification/email) isolated so they never fail the request; cron-precomputed recommendation cache honoring a stated design rule. Against: in-process cron; no service layer (controllers own business logic — fine at this size, say so if asked); implicit mock-mode selection (security O7).

  • Production boot guards against mock keys.
  • Idempotency keys for side effects if horizontally scaled.

Against: the design docs describing all of this are deleted from the repo (only in git history); 14 collections vs the docs' 12 (notifications and purchases post-date them).

  • Restore the DB spec into docs/ and update to v5 (14 collections).
  • Document the weighted-average cost formula where finance-minded graders can find it.

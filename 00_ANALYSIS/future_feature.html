Future Feature — Smart Energy Advisor

🛒 OptiCart Appliances

Per-appliance kWh specs → estimated yearly running cost under Lebanese EDL + generator pricing, with filter/sort by total cost of ownership. Verified against the current schema and filter pipeline.

The pitch in one sentence: "OptiCart stops selling appliances by their price tag and starts selling them by what they actually cost a Lebanese household — a number no competitor shows, computed from fields we can add to our existing schema in an afternoon."

What it is

Store per-appliance energy-consumption specs (kWh/year) and show every customer the estimated yearly running cost under Lebanese electricity conditions — the EDL grid tariff plus the private generator (moteur) subscription — then let shoppers filter and sort by total cost of ownership (TCO) instead of sticker price, with a user-adjustable grid/generator hours split (state-power hours vary by region).

The calculator
yearlyCostCents = kWhPerYear × (gridShare × edlTariffCents + generatorShare × generatorTariffCents)
fiveYearTCO     = basePriceCents + 5 × yearlyCostCents

Why this feature (justification)

a · Appliances are the domain — energy is the #1 hidden purchase factor

The catalog is refrigerators, ovens, cooktops, microwaves, dishwashers (server/seed.ts — OptiCool/OptiChef lines; categories "Major/Small Appliances"). A refrigerator runs 24/7 for a decade — the sticker price is a down payment. Verified against the current model: no field captures energy data (models/Product.ts has price, variants, SEO meta — nothing else), so two $899 fridges look identical today even if one costs double to run.

b · Uniquely valuable in Lebanon

Lebanese households pay two electricity bills — EDL and a private generator subscription — and generator power costs dramatically more per kWh. For an energy-hungry appliance, cumulative generator-era running costs can rival or exceed the appliance's own price within a couple of years, which makes "cost to run" a decisive local purchase factor no regional competitor surfaces. (Delivery note: quote current EDL and generator per-kWh rates from a dated source at presentation time — tariffs change; that's exactly why they're admin-configurable constants below, never hardcoded literals.)

c · Technically cheap — verified against the current code

  • Schema: ProductSchema already demonstrates the optional-subdocument pattern with meta: MetaSchema (models/Product.ts). One honest precision: meta is SEO-focused (title/og tags), so the right move is a sibling optional subdoc rather than overloading it — still the same zero-migration pattern (existing documents simply lack the field, exactly like products without brand).
  • Variant-level nuance already exists: capacity variants are modeled (IVariant.capacity), and an optional per-variant kWhPerYearDelta would mirror the existing priceDeltaCents pattern exactly.
  • Filtering/sorting: getProducts builds a filter object and a sortOption from query params (controllers/product.ts:42-134) — additions are one filter branch and one sort branch. To keep it index-friendly (matching the existing (categoryId, basePriceCents) strategy), precompute estYearlyCostCents on save — the codebase already has both precedents: the slug pre-validate hook (models/Product.ts) and the precomputed totalCostCents on purchases (models/Purchase.ts:10-11).
  • Admin input: the product form already ships structured JSON in multipart FormData (admin/Products.tsx) — energy inputs are three more controlled fields.

d · Defensible differentiator + native upsell path

No local appliance retailer ranks by cost of ownership. The upsell mechanism already exists: the recommendation system (services/scheduler.ts engine + controllers/recommendation.ts fallback chain + productRecommendations cache) can gain one "efficient alternative" rule — same category, similar price band, lower estYearlyCostCents. "This fridge costs $120/year to run — this one $55" is the highest-margin sentence a salesperson can say in Lebanon. And with the new purchases data, staff can even see margin and efficiency together when choosing what to restock.

Risks & honest caveats (say these before you're asked)

  • Data sourcing: kWh/year comes from manufacturer spec sheets entered by the inventory manager — garbage in, garbage out; the energyClass label is the low-precision fallback badge.
  • Tariff volatility: Lebanese tariffs change — hence admin-configurable constants and a precomputed, recomputable estYearlyCostCents; the UI never does tariff math from literals.
  • Coverage: the badge/sort applies only to products carrying energySpec — the optional-field design degrades gracefully instead of blocking on a full catalog backfill.

Anticipated questions

Why not just show the energy label like EU shops do?
The label is a static class letter; our differentiator is the localized money number — EDL + generator split, adjustable to the customer's own outage hours — and the ability to sort the whole catalog by it.
Why precompute instead of calculating per request?
Sorting the catalog by a computed expression can't use an index; precomputing on save (and on tariff change via the nightly cron) makes TCO sorting exactly as cheap as today's price sorting — the same reasoning the codebase already applies to recommendations and totalCostCents on purchases.
How long would this take?
Steps 1–2 are an afternoon (one optional subdoc + two query branches on an existing pipeline); the calculator UI is the bulk of a second day; the recommendation rule is a stretch goal reusing an existing nightly job.

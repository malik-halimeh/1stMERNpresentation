Tech Decisions

🛒 OptiCart Appliances

Per technology: why it was chosen, alternatives, tradeoffs, and a one-sentence "why not X?" answer a presenter can deliver.

For each technology: why it fits this codebase (with file evidence), the main alternatives, the honest tradeoff, and a one-sentence "why not X?" a presenter can deliver verbatim. Versions from client/package.json and server/package.json.

Frontend

React 19

Why: the storefront is repeated cards/tables/forms — a component model with one-way data flow (client/src/components/ui/, 17 reusable components); Context API covers the app's three shared concerns; the course is React-centric. Alternatives: Vue, Angular, Svelte. Tradeoff: more decisions left to the team than Angular; larger runtime than Svelte.

Why not Vue or Angular?
"React has the largest ecosystem and hiring pool, and its one-way data flow maps cleanly onto our catalog→card→action component tree — Angular's full framework would be overhead for a storefront this size."

Vite 8

Why: instant dev server + HMR for a 40+-component SPA; npm run build = tsc -b && vite build gives type-checking plus optimized bundles. Alternatives: Next.js, Webpack, CRA (dead). Tradeoff: no SSR — product pages are client-rendered even though the Product model carries SEO meta fields that only pay off with SSR.

Why not Next.js?
"We ship a separate Express API with its own auth, cron and webhook concerns — Next's server layer would duplicate our backend; Vite gives us the fast SPA toolchain without merging the two."

TypeScript (client ~6.0, server ^5.4)

Why: shared shape discipline — the role union appears identically in middleware/auth.ts and AuthContext.tsx; Mongoose generics (Schema<IProduct>) catch schema/interface drift at compile time; the build doubles as the regression net (both workspaces compile clean). Tradeoff: build step + a few as any escape hatches on middleware typings.

Why not plain JavaScript?
"With 14 collections and ~60 endpoints, the compiler is the only thing that tells us a renamed field broke five files — before the demo does."

Tailwind CSS 3.4

Why: a real token system in tailwind.config.js (brand palette, semantic colors, full type scale) so utilities compose one consistent design; purge keeps the bundle lean. Alternatives: CSS Modules, styled-components, MUI/Chakra. Tradeoff: long class strings; token discipline required (the config documents a bug where undefined *-bg tokens silently rendered transparent).

Why not a component library like MUI?
"We wanted the OptiCart brand identity, not Material's — Tailwind tokens gave us a custom design system at utility-class cost."

React Router 7

Why: route guards as plain components — ProtectedRoute and StaffGate express the entire auth/role routing policy in ~40 lines of App.tsx. Tradeoff: needs host rewrite-to-index config in production; raw <a> tags bypass it (the mobile-drawer regression that was found and fixed — everything now navigates via navigate()/Link).

Why not TanStack Router?
"React Router is the course standard, and our guard logic is simple enough that its component model is the clearest expression of it."

Context API (no Redux)

Why: exactly three shared concerns exist — session (AuthContext), cart badge + wishlist ids (ShopContext), toasts (ToastContext); everything else is page-local fetch state. Tradeoff: no request caching/dedup — pages refetch on mount; a context change re-renders all consumers.

Why not Redux?
"Redux earns its boilerplate when many state slices interact; we have three independent contexts, so Redux would be ceremony without benefit."

Axios

Why: the whole token strategy lives in two interceptors — Bearer attachment and the single-flight 401→refresh→retry (services/api.ts); withCredentials carries the refresh cookie. Tradeoff: one extra dependency over fetch.

Why not fetch?
"Fetch has no interceptors — we'd hand-roll the refresh-retry logic around every call; with Axios it's written once and every request inherits it."

Backend

Express 4

Why: the middleware chain mirrors the security model 1:1 — helmet → CORS → raw-webhook mount → parsers → rate limit → routers → error handler (server.ts); per-route composition keeps policy visible where the route is declared (authenticate, authorize, multer, controller). Alternatives: Fastify, NestJS, Koa. Tradeoff: no built-in structure/DI — the team imposes its own layering; slower than Fastify in benchmarks that don't matter at this scale.

Why not NestJS?
"Nest's decorators and DI shine on large teams with many modules; our 16 routers with explicit middleware chains are easier to audit line-by-line for a security-graded project."

Mongoose 8 + MongoDB

Why: the domain is document-shaped — variants embedded in products, snapshot items embedded in orders, statusHistory as an array; schema-level integrity where it matters: unique compound review index, partial unique low-stock index, append-only audit hooks, TTL on events, text search index. The purchases feature even computes weighted-average cost inside a single document update via an aggregation pipeline (controllers/purchase.ts). Tradeoff: no DB-enforced foreign keys (referential rules live in controllers, e.g. product delete blocked while in active orders); multi-document transactions need a replica set (the review code carries a standalone fallback).

Why not PostgreSQL?
"Our hot documents — a product with its variants, an order with its snapshot items — are read whole in one query; in SQL each would be three or four joins, and our money math is integer cents, not SQL aggregates, so we'd gain little."

JWT access token + rotating opaque refresh token

Why: 15-minute stateless access tokens keep every API hop cheap; the refresh token is stateful by design (SHA-256 hash on the user) so sessions are revocable and theft is detectable — atomic rotation, a 30-s multi-tab grace window, and compromise-triggered global revocation (controllers/auth.ts). Tradeoff: the most intricate code in the app (the client's single-flight logic exists solely to avoid tripping our own reuse detection); access tokens can't be revoked mid-lifetime (worst case 15 min of stale role).

Why not plain server sessions?
"Sessions put a DB read on every request and tie us to sticky state; our hybrid keeps requests stateless while the rotating refresh hash gives us the revocation and theft-detection that pure JWTs lack."

bcryptjs (cost 12)

Why: adaptive salted hashing everywhere passwords are written; the pure-JS package installs with zero native toolchain — a real concern on the team's mixed Windows machines. Tradeoff: ~30% slower than native bcrypt; argon2 is OWASP's first pick on paper.

Why not argon2?
"Argon2 is the stronger KDF on paper, but bcrypt at cost 12 is OWASP-acceptable and bcryptjs installs everywhere without a compiler — which mattered for a three-person student team."

Zod

Why: declarative validation whose field-level errors feed straight into the AppError envelope; the user-update validator doubles as a security allowlist — passwordHash/refreshTokenHash are unreachable and role/isActive are forced through dedicated audited endpoints (validators/user.ts). Tradeoff: only auth and user inputs are Zod-validated; product/coupon/cart/purchase controllers validate by hand — inconsistent depth, honestly acknowledged.

Why not Joi?
"Zod infers the TypeScript type from the schema, so the validator and the type can never drift — with Joi both must be maintained separately."

Multer 2 (memory) + Cloudinary 2

Why: buffers stream straight to Cloudinary's upload_stream — no temp files, nothing to clean; 5 MB and field-count caps at the router; CDN delivery with stored publicId for future deletion; full mock mode keeps dev offline. Tradeoff: vendor dependency; images transit our server RAM (vs presigned direct upload).

Why not store images locally?
"Local files die with the server and don't CDN; Cloudinary gives storage, transformation and CDN in one free-tier service — and mock mode keeps development offline-friendly."

Stripe 15 (PaymentIntents + webhook)

Why: PCI burden stays with Stripe — the card form is Stripe's own element; our server only ever sees a PaymentIntent id; the webhook is the single source of payment truth, received on a raw-body route and verified fail-closed (server.ts + controllers/order.ts); refunds go back through stripe.refunds.create; a keyless mock mode makes the entire flow demoable. Tradeoff: webhook correctness is on us — which is exactly where the audit found (and the team fixed) the raw-body pitfall.

Why not PayPal?
"Stripe's PaymentIntent + webhook model matches our two-phase design — capture funds, staff confirm fulfilment — and its test tooling is the best in class for a graded demo."

node-cron 4

Why: one idempotent nightly job (02:00) recomputes recommendations and trending flags; the design rule "never compute recommendations at request time" needs a scheduler, not a queue. Tradeoff: in-process — multiple instances would double-run it; a crash at 02:00 skips a day; no retry/backoff.

Why not BullMQ?
"A Redis-backed queue is the right answer at scale, but for one idempotent nightly aggregation an in-process cron is zero extra infrastructure."

Helmet 7 & CORS

Why: one-line secure-header baseline; CORS locked to a single origin from CLIENT_URL with credentials: true, trailing slash defensively stripped because browsers match Access-Control-Allow-Origin exactly (server.ts — the comment documents the real bug class).

Why not hand-rolled headers?
"Helmet encodes years of header best practice we'd otherwise have to track release by release."

express-rate-limit 8

Why: three scoped limiters (middleware/rateLimiter.ts): auth 10 failed/15 min with skipSuccessfulRequests (account switching never burns budget), coupon-apply 20/15 min keyed by userId, global 1000/15 min. Deliberate placement: /refresh and /logout are exempt because interceptors fire them automatically. Tradeoff: in-memory store — resets on restart, not shared across instances.

Why not a WAF or gateway limit?
"App-level limiting lets us key by userId and skip successful logins — semantics a generic gateway can't express."

npm workspaces monorepo

Why: one repo, two packages, root scripts, one Prettier config — atomic commits across client+server whenever an API shape changes (which is every feature here). Tradeoff: no task graph/caching (irrelevant at this size).

Why not separate repos?
"Every feature touches both sides — one commit that changes the endpoint and the page that calls it beats two coordinated PRs every time."

nodemailer 9 · morgan

nodemailer: SMTP-agnostic (Gmail app-password or any host) with a console-stub fallback so email flows demo without credentials; swapping providers is an env-var change. morgan: dev request logging — production would want structured logs (pino), listed as an improvement.

Why not SendGrid's SDK?
"Nodemailer keeps us provider-neutral — changing providers is configuration, not code."

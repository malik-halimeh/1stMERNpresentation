Architecture Report

🛒 OptiCart Appliances

Layered architecture, full request lifecycle, complete auth flow, error pipeline, DB resilience, scheduler, mailer, Stripe and Cloudinary flows.

Layered architecture

Request path top→down; scheduler and mailer are side channels called from controllers/jobs, never from the request chain. Mermaid source: diagrams/architecture.mmd

Request lifecycle (worked example)

GET /api/products?search=fridge:

  1. A page calls api.get('/products?…'); the request interceptor attaches Authorization: Bearer <memory token> if one exists (services/api.ts:23-32) — this endpoint is public, so it also works without one.
  2. Express runs the global middleware stack; the rate limiter counts the request (server.ts).
  3. routes/index.ts dispatches to the product router → getProducts.
  4. The controller builds a filter object (category slug→children expansion, dollar→cents price conversion, $text search), runs countDocuments + a paged find, and responds {success, data, meta:{total,page,limit,pages}} (controllers/product.ts:42-134).
  5. Any thrown error reaches next(error) → errorHandler → the uniform envelope {success:false, error:{code,message,details}}.

Complete auth flow

Login → session

POST /auth/login (controllers/auth.ts): Zod validation → bcrypt compare (cost 12) → isActive check → email-verified check (unverified users get a fresh code + 403) → respondWithSession:

  • signs a 15-minute JWT access token (utils/tokens.ts) — the client keeps it in a module variable, never in storage (services/api.ts:6-11);
  • generates an opaque refresh token (crypto.randomBytes(40)), stores only its SHA-256 hash on the user;
  • sets cookie refreshToken = "<userId>:<token>" — httpOnly, secure in production, sameSite:'strict', 7 days.

Single-flight refresh · rotation · reuse detection

server/src/controllers/auth.ts — the atomic rotation core
// Happy path — atomic rotation: the hash match is part of the query, so
// of N concurrent refreshes carrying the same token exactly one rotates.
const rotated = await User.findOneAndUpdate(
  { _id: userId, refreshTokenHash: incomingHash },
  {
    refreshTokenHash: hashString(newRefreshToken),
    prevRefreshTokenHash: incomingHash,
    prevRefreshTokenExpiresAt: new Date(Date.now() + ROTATION_GRACE_MS), // 30 s
  },
  { new: true }
);
  • Grace path: a token that was just rotated away by a concurrent tab is still accepted for a fresh access token (no re-rotation) within the 30-second window.
  • Reuse detection: any other presentation of an old token nulls every refresh hash, clears the cookie, and returns 403 AUTH_SESSION_COMPROMISED — full re-login required.
  • Client single-flight: one shared refreshPromise (services/api.ts:38-66) so the mount-time silent refresh, StrictMode double-mount, and any number of concurrent 401 retries all await the same rotation — a second parallel refresh would trip the server's own reuse detection.

Silent refresh · 401 retry · RBAC

  • Silent refresh on mount (AuthContext.tsx): skipped for guests via a localStorage hint (avoids a doomed 401 per anonymous page load); a sessionEpochRef counter makes stale refresh results no-ops so a slow failing refresh can't wipe a session a login just created.
  • 401 retry interceptor (services/api.ts:69-103): on 401 (excluding auth endpoints, _retry-guarded) awaits the shared refresh and replays the original request; on refresh failure, hard-redirects to /login?redirect=<path>.
  • RBAC: authenticate verifies the Bearer JWT into req.user {userId, role}; JWT errors map to distinct 401 codes in the error handler; authorize(...roles) returns 403 AUTH_FORBIDDEN. Every feature router declares its roles inline.

Password reset (code flow)

forgot-password emails a 6-digit code (hash + 15-min TTL on the user; identical response whether the account exists — anti-enumeration); reset-password validates email + code + password policy via Zod, re-hashes, clears the code, marks the email verified, and opens a session through respondWithSession — which rotates the refresh token and thereby revokes any session an attacker held before the password change (controllers/auth.ts resetPassword).

Error-handling pipeline

AppError(code, message, statusCode, details[]) (utils/errors.ts) is thrown by controllers → errorHandler (middleware/error.ts) maps AppError / Mongoose ValidationError (400) / JsonWebTokenError + TokenExpiredError (401) / everything else (500) into {success:false, error:{code,message,details}}; stacks log only outside production. notFound converts unmatched routes to 404 AppErrors. The client mirror utils/apiError.ts unwraps the envelope for toasts. Every controller — including user.ts — routes failures through this pipeline; no handler responds with raw error objects.

Database connection resilience

config/db.ts: maxPoolSize: 10, autoIndex: true; on disconnected, up to 5 retries at 5-second intervals, counter reset on success, then process.exit(1) — fail-fast so a supervisor restarts the process. The server listens only after the initial connect resolves, and boot also runs LowStockAlert.syncIndexes() to drop the replaced stale index (autoIndex only creates, never removes — server.ts).

Scheduler

One daily node-cron job at 02:00 (services/scheduler.ts, registered at startup):

  • A — Co-occurrence engine: loads purchase ProductEvents with a userId, groups per user, dedupes per product keeping the best recency weight 1/(daysSince+1), accumulates pair weights across users, upserts the top-10 neighbours per product into the cache. Reads at request time come exclusively from that cache.
  • B — Trending flags: 7-day windows — top-8 by ordered quantity → isMostSelling; top-8 by view events → isTrending (reset-then-set).
Stated scaling limit: the cron runs in-process — multiple API instances would double-run it, and a crash at 02:00 skips a day. Acceptable for one idempotent nightly aggregation; a queue is the growth path (see project_audit).

Mailer

services/mailer.ts builds its transport lazily on first send — deliberate, because ESM imports hoist above dotenv.config(), so a module-load env read would miss .env (the same pitfall that was found and fixed in utils/tokens.ts). Priority: Gmail app-password → generic SMTP → console stub. Five message types (signup code, reset code, order confirmation, status change + reason, low-stock to all active managers); all sends are fire-and-forget with logged failures.

Stripe flow

  1. Session: POST /orders/checkout-session snapshots the cart with live stock checks, fully re-validates any coupon (incl. usage caps), creates a PaymentIntent (or mock ids when keyless), persists a draft pending order whose items snapshot unitCostCents for margin math.
  2. Payment: real mode renders Stripe's PaymentElement; mock mode renders MockPaymentForm, which itself posts the fake payment_intent.succeeded event (Checkout.tsx).
  3. Webhook — fail-closed: the route receives the raw Buffer (mounted with express.raw before express.json, server.ts). With a real key + STRIPE_WEBHOOK_SECRET, a missing or invalid signature is rejected with 400 (WEBHOOK_SIGNATURE_MISSING/INVALID) — an unsigned payload can never mark an order paid. Mock/dev mode parses the JSON buffer as-is.
  4. On success (idempotent via the paymentStatus guard): atomic conditional stock decrement ($inc guarded by stock ≥ qty; lost races clamp to 0 with an oversell warning), coupon usedBy increment, cart clear, purchase ProductEvents, confirmation email. The order stays pending until staff confirm — payment captures funds; fulfilment is a human decision.
  5. Refund: status → refunded calls stripe.refunds.create (mock id in mock mode) before persisting, fills order.refund, and writes a refund_decision audit; failure aborts with 502 REFUND_FAILED.

Cloudinary flow

Multer memory storage (5 MB/file; images ≤5, variantImages ≤20 — routes/product.ts); buffers stream through cloudinary.uploader.upload_stream (config/cloudinary.ts) storing {url: secure_url, publicId}; per-variant photos match variants by an imageSlot index in the variant JSON. Without CLOUDINARY_URL the service returns deterministic mock URLs so the whole flow works offline. JSON-only PATCH requests pass through Multer untouched.

Purchases — the atomic pipeline update

The newest subsystem (controllers/purchase.ts) solves a classic inventory problem — adding stock while recomputing the weighted-average unit cost — without a transaction, using a single aggregation-pipeline findOneAndUpdate:

server/src/controllers/purchase.ts — one document operation, no read-then-write race
// newCost = (oldStock*oldCost + qty*unitCost) / (oldStock + qty), computed
// INSIDE MongoDB via a $map over variants (the positional $ can't do math).
// { new: false } returns the PRE-update doc → before/after audit for free.
const preUpdate = await Product.findOneAndUpdate(
  { _id: productId, 'variants.sku': sku },
  [ { $set: { variants: { $map: { /* $cond on sku → $mergeObjects:
        stock: $add, costPriceCents: weighted average with $round */ } } } } ],
  { new: false }
);

A compensating revert restores stock/cost if the Purchase insert fails afterwards; deletion reverts stock with the same guarded $gte pattern as the webhook and refuses (409) when stock would go negative. Cost is intentionally not reverted on delete — a weighted average is not invertible, and the comment says so.

Frontend architecture notes

  • State: Context API only — Auth (session), Shop (cart badge + wishlist ids with guest/auth branching), Toast. Pages own their fetch state; no query cache.
  • Guest commerce: localStorage cart/wishlist snapshots; utils/guestMerge.ts posts both merges after any successful sign-in, then clears local copies; the server resolves quantity conflicts with max(quantity) capped at stock.
  • Routing: public storefront, ProtectedRoute for customer pages, per-path admin routes under AdminContentRoute, StaffGate keeping staff on /admin/*, scroll-to-top + fade-slide transition per navigation, catch-all → home (App.tsx).
  • Design system: Tailwind token theme (tailwind.config.js) + 17 reusable ui/ components; lucide icons (with hand-inlined SVG brand glyphs in the footer since lucide dropped them).

1 · Authentication & sessions COMPLETE

Register → verify → login → refresh rotation → logout → password reset Malik Haya

Frontend: client/src/pages/Login.tsx, Register.tsx, ForgotPassword.tsx, context/AuthContext.tsx, services/api.ts, ui/GoogleSignInButton.tsx · API: POST /api/auth/register|verify-email|resend-verification|google|login|forgot-password|reset-password|refresh|logout, GET/PATCH /api/auth/profile · Models: User

  • Register creates an unverified account; a 6-digit code (SHA-256-hashed, 15-min TTL) is emailed; the session opens only after verify-email (controllers/auth.ts:27-34,130-168).
  • Access token = 15-min JWT held in memory only (services/api.ts:6-11); refresh token = opaque 40-byte random, SHA-256 hash stored, sent as an HTTP-only SameSite=Strict cookie, 7-day TTL.
  • Rotation + reuse detection: atomic findOneAndUpdate keyed on the old hash (exactly one of N concurrent refreshes rotates), 30-s grace via prevRefreshTokenHash for multi-tab races, reuse outside grace revokes everything → 403 AUTH_SESSION_COMPROMISED.
  • Client: single-flight refresh promise, 401 retry interceptor, silent refresh on mount gated by a localStorage hint, epoch guard so a slow failing refresh can't wipe a fresh login (AuthContext.tsx).
  • Password reset — COMPLETE (was PARTIAL in the old analysis): two-step 6-digit code flow (controllers/auth.ts resetPassword, ForgotPassword.tsx); completing it re-hashes the password, marks the email verified, and rotates the refresh token so any hijacked session dies with the old password.
  • Post-sign-in landing: customers go to the storefront home (a ?redirect= pointing at an auth page is ignored — Login.tsx:24-31); staff always land on /admin/dashboard.

9 · Checkout & payments (Stripe) COMPLETE

PaymentIntent → fail-closed webhook → atomic stock Malik Mahmod

Flow: cart snapshot with live stock checks → coupon validation → Stripe PaymentIntent (or mock ids without a key) → draft pending order → webhook payment_intent.succeeded → idempotent (paymentStatus guard) stock decrement, coupon count, cart clear, purchase ProductEvents, confirmation email. The order stays pending until staff confirm — a deliberate two-phase design (controllers/order.ts).

  • FIXED Webhook signature verification (old analysis: structurally broken): the route is mounted with express.raw before express.json (server.ts:30-34) so constructEvent gets the exact bytes; with a real key + webhook secret the handler fails closed — 400 WEBHOOK_SIGNATURE_MISSING/INVALID for unsigned or forged payloads. Mock mode still parses the JSON buffer so the demo MockPaymentForm works. Verified by test_fixes.ts checks 3–4 and an HTTP smoke test through the real middleware.
  • FIXED Oversell race: stock decrement is an atomic conditional $inc (stock: {$gte: qty} in the query); a lost race clamps to 0 with an explicit oversell warning.
  • Remaining nuance (documented, accepted): orderNumber is a random 6-digit string — the unique index turns a collision into an error rather than a silent duplicate; no retry loop exists.

11 · Stock purchases / procurement COMPLETE NEW

Weighted-average cost, atomic pipeline update, revert guard Malik Mahmod Haya

Frontend: admin/Purchases.tsx (387) + sidebar item + dashboard month-spend tile + analytics spend chart · API: GET /api/purchases/summary, GET/POST /api/purchases, DELETE /:id (staff) · Models: Purchase (14th collection), Product, AuditLog, LowStockAlert

  • Recording a purchase adds stock and recomputes the variant's weighted-average costPriceCents in a single aggregation-pipeline findOneAndUpdate — no read-then-write race with concurrent sales (controllers/purchase.ts:83-139); {new:false} returns the pre-update doc so the before/after audit needs no second read.
  • If the Purchase insert fails after the stock update landed, a compensating revert restores stock and cost (documented tiny non-transactional window, consistent with the codebase's single-document approach).
  • Deleting a purchase reverts stock with the same guarded $gte pattern as the webhook — 409 if the revert would drive stock negative; cost price is deliberately NOT reverted (weighted average is not invertible), stated in the code comment.
  • Both create and delete write stock_purchase audit entries; month-to-date KPIs via an aggregation (getPurchaseSummary); spend feeds the new /api/analytics/purchase-spend series.

15 · Recommendations engine COMPLETE

Nightly co-occurrence + trending flags, cache-only reads Malik Mahmod

16 · Product event tracking COMPLETE — with a stated limitation

Batched anonymous views, TTL-bounded Mahmod

Client buffers view events and posts ≤50 per batch to the public /api/product-events/batch; entries are validated and TTL-expired after 180 days. Limitation (by construction, verified still present): the route has no authenticate, so client view events always store userId: null — only webhook-written purchase events carry a userId, which is exactly what the co-occurrence engine consumes. Per-user view history therefore doesn't exist, and view counts are spoofable (bounded by the batch cap and the global rate limit).

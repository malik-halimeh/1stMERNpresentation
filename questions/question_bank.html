1 · Architecture & system design

Malik Why a monorepo?
One repo, two npm packages (client/server) sharing tooling and types, one place to clone and run. It keeps frontend and backend in lockstep without the overhead of separate repos.
Malik Why MERN specifically?
One language (TypeScript) top to bottom lowers context-switching, Mongo's document model fits product-with-variants naturally, and the ecosystem is mature. It let a three-person team move fast without gluing mismatched stacks.
Malik Walk me through what happens when a request arrives.
helmet → cors → (raw body for the webhook path, else json) → morgan → rate limit → per-route authenticate/authorize → controller → either a JSON response or the single errorHandler envelope.
Malik Why does middleware order matter?
Because the Stripe webhook needs the unparsed body — its raw mount must precede express.json(), or the signature can't be verified. Security headers come first so they apply to everything.
Malik How is the code organised on the server?
By layer: routes (wiring) → controllers (logic) → models (data) → services (cron, mail, recommendations), with middleware and validators cross-cutting. Each domain has its own controller and router.
Haya How is the frontend organised?
Pages, a reusable ui/ library, two layouts (storefront/admin), three contexts for shared state, and one Axios service. Routes and guards live in App.tsx.
Malik Where does business logic live — client or server?
Server, always. The client is a presentation layer; prices, coupons, stock, and authz are all decided server-side so a tampered client changes nothing that matters.
Malik How do the three of you avoid stepping on each other?
Ownership boundaries: Haya in client/, Mahmod in models/ + indexes, Malik across the rest of the server. The layered structure means changes rarely collide.
Malik Is this stateless or stateful?
The API is stateless per request (authz rides in the JWT); the only server-side session state is the hashed refresh token in Mongo and the in-memory rate-limit/cron, which are the single-instance caveats.
Malik How would you scale this to multiple instances?
Move the rate-limit store to Redis and add a cron leader-lock so the nightly job runs once. Orders/payments already scale because they're database-atomic.
Mahmod Why MongoDB over a relational DB?
Products with embedded variants, orders with line-item snapshots, and flexible catalog attributes map cleanly to documents. We'd reach for SQL if the domain were heavily relational with strict joins — it isn't here.
Malik What are the external dependencies and what happens if one is down?
Stripe (payments), Cloudinary (images), SMTP (email). Email degrades to console logging; a Stripe/Cloudinary outage fails the specific operation with a typed error, not the whole app.
Malik How is configuration managed?
Environment variables via .env (with a committed server/.env.example); production refuses to boot without critical secrets like JWT_ACCESS_SECRET.
Malik What's the single biggest architectural strength?
Correctness under concurrency: atomic stock, atomic weighted-average costing, fail-closed payments, and rotation-with-reuse-detection auth. The money paths are hard to break.
Malik Biggest architectural weakness?
Single-instance assumptions (cron + rate limit) and no CI/Docker. It's built to run correctly on one box, not yet to scale horizontally.
Malik Why TypeScript everywhere?
Types catch API/shape mismatches at compile time across the client/server boundary, which is exactly where runtime bugs love to hide.
Malik How many collections and why that many?
14, one per bounded concept (users, products, orders, reviews, coupons, purchases, audit, …). Each earns its place; none is a catch-all.
Malik If you rebuilt it, what would you change?
Add CI + Docker from day one, split the largest controllers into service functions, and design for multi-instance (Redis, leader-lock) up front.

2 · Authentication & security

Malik Describe your auth model in two sentences.
A 15-minute JWT access token kept in memory for cheap authorization, plus a 7-day opaque refresh token in an httpOnly SameSite=Strict cookie, stored only as a SHA-256 hash. Rotation on every refresh with a grace window and reuse detection.
Malik Why not just one long-lived token?
A long-lived token that's stolen is a long-lived breach. Short access + rotating refresh limits the damage window to minutes and lets us revoke.
Malik Why is the refresh token opaque, not a JWT?
So we can revoke it instantly by nulling the stored hash. A JWT can't be un-issued without a denylist, which reintroduces server state anyway.
Malik What stops XSS from stealing the session?
The refresh token is httpOnly — JavaScript can't read it. The access token is in memory and dies on reload; it's never persisted where a script can grab it.
Malik What stops CSRF?
SameSite=Strict on the refresh cookie means the browser won't send it on cross-site requests, and state-changing calls require the bearer access token that a cross-site page can't read.
Malik Explain reuse detection.
If an old (already-rotated) refresh token is presented outside the 30-second grace, that's a replay — we treat it as theft and revoke every session for that user.
Malik What's the grace window for?
Two tabs can refresh nearly simultaneously; the one arriving within 30s matching the previous hash still succeeds, so legitimate concurrency doesn't trigger the theft response.
Malik How are passwords stored?
bcrypt at cost 12 via a pre-save hook — plaintext never reaches the database, and the cost makes brute force expensive.
Malik How does password reset work?
A 6-digit code is issued and emailed (console in dev); verifying it re-hashes the new password, confirms the email, and rotates the refresh token to kill any hijacked session.
Malik Is there rate limiting on auth?
Yes — a stricter limiter on auth routes on top of the global 1000/15-min ceiling, to blunt credential stuffing.
Malik How is RBAC enforced?
authenticate verifies the JWT and sets req.user; authorize(...roles) gates each route. The client mirrors it for UX but the server is the boundary.
Malik A user's role changes — how fast does it take effect?
Up to 15 minutes, because the role is in the access token until it refreshes. That's the accepted cost of stateless authz; instant revocation would need a per-request store lookup.
Malik How do you validate input?
Zod schemas as allow-lists strip unknown fields, so a client can't smuggle role or isActive into a create/update.
Malik Do you leak whether an email exists?
On register, yes — a duplicate returns AUTH_EMAIL_IN_USE. It's a deliberate UX tradeoff we can name; login errors stay generic to avoid an oracle.
Malik Why store only the hash of the refresh token?
A DB leak then hands out no usable sessions — same principle as hashing passwords.
Malik How is the JWT secret handled?
Read lazily so .env is honoured; production won't boot without it; dev has a loud-warning fallback. An earlier version always used the fallback due to import hoisting — that's fixed.
Malik What security headers are set?
helmet applies a sensible default set (HSTS, no-sniff, frameguard, etc.) before any route runs.
Malik What's your biggest remaining security caveat?
The single-instance rate-limit store and role staleness are the known ones; both are documented, bounded, and have clear fixes.

3 · Backend & API

Malik How are errors handled consistently?
Controllers throw AppError(message, status, code); one errorHandler serialises everything to { error: { message, code } }. The client always gets the same shape.
Malik Why typed error codes and not just messages?
Codes are a stable contract the client can branch on and localise; messages are for humans and can change without breaking clients.
Malik How do image uploads work?
Multer buffers files in memory (no disk), then they're streamed to Cloudinary; only the returned secure URLs are stored on the product.
Malik Why memory storage for Multer?
No temp files to clean up or leak, and streaming straight to Cloudinary keeps the server stateless about files.
Malik How does the recommendation engine work?
A nightly node-cron job computes recency-weighted product co-occurrence from purchase events and upserts top-10 lists; requests only read the cache.
Malik Why precompute recommendations?
Co-occurrence over history is expensive; nightly precomputation keeps request latency flat, and recommendations don't need to be second-fresh.
Malik What happens on a cold start with no data?
A fallback returns popular items so new products/users still get sensible recommendations.
Malik How does email sending work?
services/mailer.ts sends via SMTP if configured, else logs to console — so demos need zero mail config. Five triggers total.
Malik What are the analytics endpoints?
Five aggregation series — revenue, order volume, top SKUs, customer growth, and purchase spend — computed with Mongo pipelines over delivered orders and registrations.
Malik How is pagination done?
Query params (page/limit) with a meta block in the response (total, page, pages), so the client can render controls without guessing.
Malik How is the audit log written?
Privileged controllers write an append-only entry (actor, action type, target, before/after) — 9 action types, never updated or deleted.
Malik Why append-only for audit?
An audit trail you can edit is worthless. Immutability is the whole point — it's accountability.
Malik How do you prevent trusting client prices?
Checkout re-prices every line from the database and recomputes the coupon; the client's numbers are display-only.
Malik How are long-running jobs scheduled?
node-cron inside the process starts after the DB is up. It's simple and correct for one instance; multi-instance would need a leader-lock.
Malik What does the health endpoint do?
Returns a basic liveness signal so a load balancer or uptime check can tell the process is up.
Malik How do you keep controllers testable?
Logic funnels through typed errors and pure-ish helpers (e.g. weighted-average cost), and there's a regression suite (test_fixes.ts, 20 checks) covering the fixed paths.
Malik Why 16 routers instead of one big file?
One router per domain keeps concerns isolated and the wiring readable; server.ts just mounts them under /api/*.
Malik How is CORS configured?
Explicit origin allow-list with credentials enabled, because the httpOnly refresh cookie must travel cross-origin.

4 · Database & data modeling

Mahmod When do you embed vs reference?
Embed what's owned and read together (variants in a product, addresses in a user); reference what's shared and edited independently (orders → products).
Mahmod Why snapshot product name/price on orders?
So renaming or repricing a product later never rewrites past orders. History must be immutable.
Mahmod What's denormalized and why?
ratingAvg/reviewCount on products, so listing pages read O(1) instead of aggregating reviews on every render.
Mahmod How do you keep denormalized ratings correct?
A transaction recomputes them on every review create/edit/delete — the write pays so the read stays cheap and consistent.
Mahmod Explain the partial unique index.
Unique on (productId, variantSku) only where status:'active' — one active low-stock alert per variant, unlimited resolved history.
Mahmod What's a TTL index used for here?
Product view events carry a 180-day TTL so the stream self-prunes; no cleanup job, no unbounded growth.
Mahmod How does search work at the DB level?
A text index over product name/description backs the search query so Mongo ranks matches instead of scanning.
Mahmod Why compound indexes on the audit log?
It's queried by actor, action type, and target; indexing those keeps "everything this admin did" fast as the log grows.
Mahmod How is one-review-per-user enforced?
A unique (user, product) index — the database rejects a second review, no app-level race can slip past it.
Mahmod How is money represented?
Integer cents everywhere. Floats can't represent currency exactly, so we never use them for money.
Mahmod How is the category tree modeled?
Self-referencing parent, two levels in practice, with a unique slug for URLs and filtering.
Mahmod What does syncIndexes do at boot?
Reconciles live indexes with the schema — it's how the old full unique index on alerts was replaced by the partial one with no manual migration.
Mahmod Explain weighted-average costing.
New cost = (oldStock·oldCost + qty·newCost)/(oldStock+qty), computed atomically inside the purchase update so stock and cost never disagree.
Mahmod Why snapshots on the Purchase model?
Product name and buyer name are copied so procurement records stay readable even if the product or user changes later.
Mahmod Do you use transactions?
Yes, on the review flow, so the Review write and the product rating recompute commit together or not at all.
Mahmod How does the seed script stay safe to re-run?
It's idempotent — upserting known fixtures rather than blindly inserting, so re-seeding doesn't duplicate data.
Mahmod What's your migration story?
syncIndexes handles index drift; schema data-migrations are manual today — a formal framework is a known gap.
Mahmod Where would this schema strain first at scale?
Order and ProductEvent growth; I'd shard Order by customer/time and roll events into daily aggregates. Embedding actually reduces cross-shard joins.

5 · Frontend & UX

Haya Why Context instead of Redux?
Three bounded slices (auth, shop, toast) fit Context cleanly without Redux boilerplate. It's the right size for the state we actually share.
Haya Where is the access token stored client-side?
In memory in AuthContext — never localStorage — and silently refreshed from the httpOnly cookie on reload.
Haya What is single-flight refresh?
Concurrent 401s share one refresh promise, so ten failures cause one refresh and ten replays — not ten refreshes racing each other.
Haya Why one Axios instance?
baseURL, credentials, and the refresh interceptor are configured once and inherited by every call — no inconsistent fetch handling.
Haya How do guest carts survive login?
Guests keep cart/wishlist in localStorage; on login the client merges into the server cart and clears local, with stock re-validated server-side.
Haya How do route guards work and are they secure?
ProtectedRoute gates by role for UX; it's not security — the server authorises every request, so bypassing the guard just yields a 403.
Haya Where does a user land after login?
Customers on the storefront, staff on the admin dashboard; a redirect back to an auth page is ignored so nobody is stranded.
Haya How is filtering/search state managed?
In the URL query string, so results are shareable, bookmarkable, and back-button-correct; the list page reads params and refetches.
Haya Why is search a submit form now?
The header search submits to the product list with ?search= — simpler and fully accessible; the earlier live-suggestion dropdown was removed.
Haya How do you handle loading states?
Skeleton loaders so layout doesn't jump, plus disabled buttons during mutations — the user always sees progress.
Haya How do product variants work in the UI?
Selecting a variant swaps price, gallery, and stock, and add-to-cart targets that variant's SKU specifically.
Haya What's the DataTable component?
A reusable admin table (sort, paginate, row actions) used across Products, Orders, Coupons, Users and more for consistent behaviour.
Haya How is the design system kept consistent?
Tailwind tokens defined once in the config; components consume tokens, so a theme change is one edit.
Haya Is the app responsive?
Yes, mobile-first: header collapses to a drawer, admin sidebar collapses, grids reflow. A responsiveness pass fixed the earlier full-reload mobile drawer.
Haya How does the app show money?
The server sends cents; the UI formats to currency at render only, never doing math on displayed dollars.
Haya How does the notification bell work?
Shows an unread count with a dropdown and mark-all-read, reading the per-user Notification collection through the API.
Haya Why is the bundle 546 kB and what would you do?
It's one chunk; I'd route-split the admin area from the storefront with lazy imports since most visitors never load admin.
Haya How does TypeScript help the UI?
Shared response types make API mismatches compile errors, and typed props catch wrong usage before runtime.

6 · Payments, orders & inventory

Malik Walk me through a checkout.
Server re-prices the cart and re-validates the coupon, creates a Stripe PaymentIntent, Stripe calls the fail-closed webhook, stock decrements atomically, and the order is created from server prices with snapshots.
Malik What does fail-closed mean for the webhook?
A missing or invalid signature is rejected with a 400 — there is no branch that trusts an unverified body. You can't fake a "payment succeeded".
Malik How do you prevent overselling?
Stock decrement is a conditional $inc that only applies when stock ≥ qty; a lost race clamps to 0 and logs — it never goes negative.
Malik Why re-validate the coupon at checkout if it was previewed?
The preview is client-controlled by the time it returns; anyone can POST a coupon straight to checkout, so the server re-checks caps, expiry, and min-order.
Malik How do refunds work?
stripe.refunds.create (mock id if keyless) runs before the status persists; the refund subdoc is filled, a refund_decision audit is written, and a Stripe failure aborts with 502.
Malik What is mock mode?
When Stripe keys are absent (dev), payments and refunds are simulated so the flow runs end to end without real charges.
Malik Could mock mode run in production?
The secret fail-fast means a misconfigured prod deploy won't boot, surfacing the missing keys before traffic. I'd still add an explicit prod assertion.
Malik Describe the order state machine.
Seven states with enforced legal transitions and recorded reasons; illegal jumps are rejected and each change is auditable.
Malik How are order totals trusted?
They're computed server-side from snapshotted line prices, never from client input.
Mahmod How does restocking affect cost?
A purchase adds stock and blends the weighted-average cost in one atomic update, so margin math stays honest as buy-prices drift.
Malik What if the Purchase record fails to save after the stock update?
A compensating update reverts the stock/cost change; deletes mirror this and refuse (409) if the revert would make stock negative.
Mahmod How do low-stock alerts fire?
A stock write crossing the threshold creates a duplicate-safe alert; the partial index guarantees one active alert per variant.
Malik Which events feed recommendations — views or purchases?
Purchase events (they carry userId); anonymous view events are supplementary and, by design, not user-attributed.
Malik How are coupons modeled?
With global and per-user usage caps plus expiry and min-order, all re-checked at checkout, and audited CRUD in admin.
Mahmod Can a customer review anything they want?
No — only a product from a delivered order (verified-purchase rule), enforced before the review is accepted.
Malik What happens to stock if a payment fails?
Nothing — stock only decrements on payment_intent.succeeded, so a failed payment leaves inventory untouched.
Haya What does the customer see after ordering?
An order confirmation rendered from the server response, and the order appears in their history with a status stepper.
Malik How is a duplicate webhook handled?
We create one order per intent, so a repeat finds it already exists; I'd add an explicit processed-event guard by event id to make idempotency bulletproof.

7 · "We actually hit this" — war stories & behavioral

Malik Tell me about a real bug you fixed. (Webhook)
Our webhook signature verification was structurally broken: express.json() consumed the raw body before Stripe could verify it, so the handler fell back to trusting unverified input. We mounted express.raw() on the webhook path before the JSON parser and made verification fail-closed. Lesson: middleware order is correctness, not cosmetics.
Malik A subtle one you're proud of catching? (JWT secret)
The JWT secret was read at module load, but ESM hoists imports — so it ran before .env applied and silently always used the fallback secret, even when a real one was set. We made the read lazy and added a production boot refusal. It taught me to respect import evaluation order.
Mahmod A database bug? (Low-stock index)
A full unique index on (productId, variantSku, status) blocked a second resolved alert per variant — resolving twice threw E11000. We switched to a partial unique index scoped to active alerts and let syncIndexes drop the old one. That's where I really learned partial indexes.
Malik A concurrency bug? (Oversell race)
Stock was decremented read-modify-save, so two simultaneous orders could both pass the check and oversell. We moved to an atomic conditional $inc guarded by stock ≥ qty, letting the database arbitrate. Correct concurrency belongs in the DB, not the app.
Malik A money bug? (Refunds)
"Refunded" was only a status change — no Stripe call, no record, no audit. We made it call stripe.refunds.create before persisting, populate the refund subdoc, write a dedicated audit, and abort on failure. Status should never claim money moved when it didn't.
Haya A frontend bug? (Stranded on login)
A ?redirect= pointing back at an auth page could loop the user on the login screen after signing in. We now ignore auth-page redirects and send customers to the storefront, staff to the dashboard. Small bug, very visible.
Haya A navigation bug? (Mega-menu)
The category mega-menu was hardcoded with names instead of slugs and fake featured ids, so links went nowhere, and the mobile drawer did full page reloads. We fed it live categories and made every click a slug-based SPA navigation.
Malik A coupon bug? (Checkout bypass)
Coupon usage and per-user caps were checked in preview but not at checkout, so you could bypass them by posting the code straight to checkout. Now checkout re-validates everything server-side.
Malik How did you verify the fixes didn't break anything?
A 20-check regression suite (server/src/test_fixes.ts), clean TypeScript builds of both packages, and an HTTP smoke test through the real middleware chain. All green.
Mahmod How did you split the work as a team?
By layer and strength: Haya on the frontend, Mahmod on models and indexes, Malik on backend logic and architecture. The Jira board (11 epics, 40 sub-tasks) reflects that split, roughly even on story points.
Mahmod The board shows checkout as In Progress — is it done?
The CSV is an early export; every epic on it is now complete and hardened in the code. Payments actually went beyond the original plan with a fail-closed webhook, atomic stock, and real refunds.
Malik What did each of you learn?
Malik: compensating transactions for multi-document safety. Haya: single-flight request coordination. Mahmod: partial indexes as business rules. Each came out of a real problem, not a tutorial.
Malik What are you honestly still not happy with?
Single-instance cron and rate limiting, no CI/Docker, server/dist tracked in git, and a 546 kB bundle. All known, all with clear fixes — none is a happy-path correctness bug.
Malik If we gave you one more week, what ships?
Redis-backed rate limiting + cron leader-lock, route-based code splitting, CI with the regression suite, and a Dockerfile. That's the gap between "correct on one box" and "production-ready".
Haya What are you most proud of as a team?
That the hard parts — auth rotation, payments, atomic inventory and costing — are genuinely correct under concurrency, and we can each explain why, not just that they work.
Malik What surprised you most building this?
How much of "security" and "correctness" is really about ordering — middleware order, do-then-undo order, verify-before-persist order. Get the order right and whole classes of bugs vanish.
